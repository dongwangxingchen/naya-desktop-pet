# 决策 0001：免费开放角色生产管线

状态：原则已确认，具体工具待技术门验证

日期：2026-07-13

## 已确认的决定

- 不购买 Live2D Cubism、Spine、Rive 等付费制作工具。
- 不依赖专业绑定师。
- Godot 4.6.3 继续负责桌宠窗口、状态机、UI、AI、记忆与平台能力。
- 正式角色生产只使用免费或开源工具，并保存可继续编辑的分层母版、绑定工程和自动检查结果。
- 不再使用不同 AI 成图互相抠取局部、拼成眨眼或表情帧。

## 尚未锁死的部分

绑定编辑器与实时角色运行时不是同一件事，必须分别验证：

1. 作者工具优先验证 `nijigenerate MCP`。它能让 Codex 通过结构化命令导入 KRA/PSD、创建网格与参数、写入绑定、截图检查、保存和透明导出。
2. Godot 运行端先保留“固定分层实时参数 + 同一绑定模型烘焙的透明动作序列”这一低风险基线。
3. `nicxlive -> Godot GDExtension` 只作为高自由度候选。没有通过加载、渲染、遮罩、性能和稳定性测试前，不能成为必需依赖。
4. 如果 `nijigenerate` 不稳定，回退到 Blender 自动化加 Godot NativeRig；Proscenio 等项目只作为可审计参考或隔离样机。

## 第一技术门

只使用临时脸部小样，不先绑定正式纳雅：

- 导入脸底、双眼、眼皮、瞳孔和前发。
- 由 Codex 创建 `EyeOpen`、`EyeXY`、网格和绑定。
- 自动截取开眼、半闭、闭眼与多个视线方向，并叠加网格检查。
- 保存、退出、重新打开后读取参数并复核。
- 导出 `.inx`、`.inp` 和透明 PNG；连续重复三轮。
- 任何崩溃、保存损坏、身体位移或闭眼穿帮都记录为失败，不在正式素材上补救。

只有小样通过后，才决定是否把 `nijigenerate` 纳入正式管线。

## 为什么这不是旧版“抠图大法”

标准分层只从同一张获批母版建立一次，并补画被遮挡的隐藏区域。眨眼只驱动固定眼皮层或同一网格的参数，身体、服装、画布和锚点保持不变。复杂动作即使烘焙为序列，也必须来自同一绑定工程，而不是让图像模型逐帧重画整个人。

## 代价与回退

- `nijigenerate` 的 MCP 目前只在 1.0 测试版中提供，可能崩溃，必须操作副本并频繁版本化保存。
- 开源 nijilive 运行时尚无现成 Godot 插件，自行接入属于中型渲染工程。
- Blender/Godot NativeRig 制作速度可能更慢，但许可证、运行时和长期维护风险最低，是必须保留的安全回退。
